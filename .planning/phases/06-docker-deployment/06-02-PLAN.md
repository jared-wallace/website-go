Create `docker-compose.yml` in the project root. This is the production compose file (per D-07: separate from docker-compose.dev.yml).

```yaml
services:
  postgres:
    image: postgres:16-alpine
    restart: unless-stopped
    environment:
      POSTGRES_USER: ${POSTGRES_USER}
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
      POSTGRES_DB: ${POSTGRES_DB}
    volumes:
      # EBS bind-mount. Host dir MUST exist with correct ownership:
      #   sudo mkdir -p /var/www/html/pgdata
      #   sudo chown 999:999 /var/www/html/pgdata
      - /var/www/html/pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U $${POSTGRES_USER}"]
      interval: 5s
      timeout: 5s
      retries: 5

  app:
    build: .
    restart: unless-stopped
    ports:
      - "8080:8080"
    env_file:
      - /var/www/html/.env
    volumes:
      # Image uploads — EBS-backed, survives container restarts
      - /var/www/html/images:/var/www/html/images
    depends_on:
      postgres:
        condition: service_healthy
    healthcheck:
      test: ["CMD-SHELL", "wget -qO- http://localhost:8080/health || exit 1"]
      interval: 10s
      timeout: 5s
      retries: 3
      start_period: 30s
```

Key design points per locked decisions:
- D-07: Separate prod file (docker-compose.yml) from dev (docker-compose.dev.yml)
- D-08: `restart: unless-stopped` on both; health checks on both (pg_isready for postgres, wget /health for app)
- D-05: `env_file: /var/www/html/.env` (absolute EBS path, never committed to git)
- No `version:` key — compose v2 ignores it, omitting avoids confusion
- `depends_on: condition: service_healthy` ensures Postgres is ready before app starts (avoids migration race)
- `start_period: 30s` gives the app time to run migrations before health checks begin failing
- Postgres env vars use `${VAR}` interpolation from the shell environment or .env — on EC2 the deploy target sources from `/var/www/html/.env`
- `$$` escaping in pg_isready health check test because compose interpolates `$` in strings — `$$` produces a literal `$` passed to the shell
- wget is pre-installed in alpine:3.21 — no extra package needed for the app health check

Extend the existing Makefile with three new targets: `deploy`, `logs`, `status`. Per D-01 and D-02, `make deploy` is the single command for production deployment via SSH.

1. Update the `.PHONY` line at the top to include the new targets:
```makefile
.PHONY: build test lint run dev dev-up dev-down migrate docker hash-password deps check-deps deploy logs status help
```

2. Add the following targets BEFORE the `## help:` target (so they appear in help output):

```makefile
## deploy: deploy to production (run on EC2 after SSH)
deploy:
	@test -f /var/www/html/.env || \
		(echo "ERROR: /var/www/html/.env not found." && \
		 echo "Copy .env.example to /var/www/html/.env and fill in values." && \
		 exit 1)
	git pull
	docker compose build --no-cache
	docker compose up -d
	@echo "Deploy complete. Use 'make logs' to watch."

## logs: tail application logs
logs:
	docker compose logs -f app

## status: show container status
status:
	docker compose ps
```

Key design points:
- D-06: `deploy` checks `/var/www/html/.env` exists before proceeding; fails with helpful message if missing
- D-02: `deploy` runs the full sequence: git pull, build, up -d — one command, no steps to forget
- `--no-cache` on build ensures a clean image on each deploy (binary is small, build time is trivial)
- `logs` targets the `app` service specifically (not postgres) for cleaner output
- `status` shows both containers via `docker compose ps`
- Uses `&&` (not `;`) in the error message so `exit 1` actually stops make
- The `@` prefix on test and echo suppresses make's own command echo for cleaner output
