Create three new files in the project root:

**Dockerfile** (multi-stage, per D-04 and FOUND-04):
```dockerfile
# Build stage
FROM golang:1.26-alpine AS builder

RUN apk add --no-cache git

WORKDIR /build

COPY go.mod go.sum ./
RUN go mod download

COPY . .

RUN CGO_ENABLED=0 GOOS=linux go build -ldflags="-s -w" -o bin/server ./cmd/server

# Runtime stage
FROM alpine:3.21

RUN apk add --no-cache ca-certificates tzdata

RUN adduser -D -u 1001 appuser
USER appuser

WORKDIR /app
COPY --from=builder /build/bin/server .

EXPOSE 8080
CMD ["./server"]
```

Key points:
- `golang:1.26-alpine` build stage, `alpine:3.21` runtime (per STACK.md)
- `CGO_ENABLED=0` for static binary (per STACK.md)
- `-ldflags="-s -w"` strips debug symbols (~30% smaller binary)
- Non-root `appuser` UID 1001 (avoids conflict with postgres UID 999)
- `ca-certificates` required for pgx TLS; `tzdata` for SQL timestamps
- No COPY of web/ directory needed in runtime — templates/static are go:embedded into the binary

**`.dockerignore`**:

Excludes secrets, build artifacts, planning docs, and dev config from the Docker build context. Note: `*.md` is safe because markdown post content is not in the repo root (it's in the DB). The go.mod and go.sum are not `.md` files so they pass through.

  <action>
Add a `/health` endpoint to the blogMux in `cmd/server/main.go`. This endpoint returns a bare HTTP 200 with no body — it exists solely for Docker's health check probe (per D-08).

Insert this line BEFORE the catch-all 404 route (`blogMux.HandleFunc("GET /{path...}", blog.NotFound)`), after the `/api/push` route registration:

```go
blogMux.HandleFunc("GET /health", func(w http.ResponseWriter, r *http.Request) {
    w.WriteHeader(http.StatusOK)
})
```

The placement is critical: Go 1.22+ ServeMux matches the most specific pattern first, but the catch-all `GET /{path...}` would match `/health` if registered before it. Place /health on the line immediately before the catch-all to keep the code readable and intention clear.

No DB ping in the health check — Postgres liveness is handled by the compose-layer `pg_isready` health check. The app health check proves the binary is alive and listening. A DB ping would cause unnecessary cascading restarts if Postgres has a brief hiccup.
