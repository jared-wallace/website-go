## Phase 5 Component Inventory

### New: Upload Button (admin-editor.html)

Placement: above the `#editor-body` textarea, inside `.admin-editor-pane`, below the `.pane-label` "Write" strip.

```
[Write label strip]
[Upload Image button]      ← new, left-aligned inside editor pane
[textarea #editor-body]
```

Styling contract:
- Element type: `<button type="button">` (not submit)
- Class: `action-link` (existing admin.css class — accent color, 44px min-height, no border)
- Label: "Upload Image"
- Loading state label: "Uploading…" (button disabled, `cursor: not-allowed`)
- Error state: inline `<span class="flash-error">` rendered below button, text per Copywriting Contract
- Success state: no visible feedback — markdown tag auto-inserted at cursor; button returns to default

The button must NOT be a `.cta-button` (that weight is reserved for primary actions like Save Draft / Publish). Use `action-link` style — same weight as Edit and Delete in the dashboard.

Source: CONTEXT.md D-01 (button placement), D-02 (button only, no drag-and-drop).

### New: Image serving (no UI)

`/images/{filename}` is a static file route. No UI component. Existing `.post-body img { max-width: 100%; height: auto; }` in main.css already handles image rendering in post bodies.

Source: CONTEXT.md D-07; main.css `.post-body img`.

### Existing unchanged: Split-pane editor layout

No structural changes to `.admin-layout`, `.admin-editor-pane`, `.admin-preview-pane`, or toolbar. Upload button slots into the existing pane without altering the grid.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Upload button (default) | Upload Image |
| Upload button (loading) | Uploading… |
| Upload error — file too large | File exceeds 5 MB limit. Choose a smaller image. |
| Upload error — wrong type | Only JPEG and PNG are accepted. |
| Upload error — server failure | Upload failed. Try again. |
| API push 401 response | unauthorized |
| API push 400 — missing slug | front matter must include 'slug' |
| API push 400 — bad request | file too large or bad request |
| API push 500 | failed to upsert post |
| API push success (200) | ok |

Notes:
- API responses are plain text (not JSON), per RESEARCH.md Pattern 4 — consistent with minimal-deps philosophy.
- Upload JS error messages render in `.flash-error` span below the upload button; they are ephemeral (cleared on next upload attempt).
- No empty state applies to the upload button — it is always present in the editor.
- No destructive confirmation required for image upload (no delete-image flow in this phase).

Source: CONTEXT.md (Claude's Discretion — error response format); RESEARCH.md Pattern 4.

## Interaction Contract

### Upload Flow (JS)

1. User clicks "Upload Image" button.
2. Hidden `<input type="file" accept="image/jpeg,image/png">` triggers file picker.
3. On file select: button label changes to "Uploading…", button disabled.
4. `fetch('POST /admin/images/upload', FormData)` — same-origin, no CORS concerns.
5. On success (200): parse JSON `{ url, markdownTag }`. Call `insertAtCursor(textarea, markdownTag)`. Dispatch `input` event to trigger live preview debounce. Button returns to "Upload Image", enabled.
6. On error: button returns to "Upload Image", enabled. Show `.flash-error` span with message per Copywriting Contract.

Debounce: upload success triggers existing preview debounce via `dispatchEvent(new Event('input'))` — no new debounce logic.

Source: RESEARCH.md Pattern 5 (`insertAtCursor`); CONTEXT.md D-01 (auto-insert at cursor).

### API Push (no UI)

`POST /api/push` is a CLI-only endpoint. No browser UI required. Bearer token auth. Plain text response. No design contract items beyond copywriting above.

Source: CONTEXT.md D-08 through D-11; RESEARCH.md Pattern 4.

## Responsive Behavior

Upload button inherits admin pane's existing responsive behavior. At `max-width: 768px`:
- Editor pane is full-width (`.admin-layout` collapses to 1-column per existing admin.css media query).
- Upload button remains in `.admin-editor-pane`, visible in Write tab.
- Upload button must not appear in Preview tab (it is DOM-sibling to the textarea, inside the editor pane — hidden automatically when editor pane is hidden via `.hidden` class from existing JS tab switching).

Source: admin.css `@media (max-width: 768px)` rules.

## Accessibility

- Upload button: `type="button"`, minimum 44x44px touch target (matches existing `action-link` min-height).
- File input: visually hidden but reachable; button click triggers input `click()` programmatically.
- Loading state: button `disabled` attribute set during upload — prevents double-submit.
- Error messages: `.flash-error` span is adjacent to the button in DOM order — screen readers encounter it naturally after the button.
- Auto-inserted markdown tag: no additional ARIA needed; textarea already has `aria-label="Markdown editor"`.

Source: existing admin.css accessibility patterns; WCAG 2.1 AA touch target minimum.
